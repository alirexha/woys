# Auto-sweep methodology — synthetic latency tuning without manual mic tests

> "User is tired of manual testing. ONE final mic test, not seven."

The v0.7.0-rc1 → rc2 cycle introduced an automated harness for sweeping
`output_latency_ms` (and any future per-chunk knob) against a synthetic
speech-like fixture. This doc describes the harness, the fixture, the
synthetic-vs-real correlation we measured, and the rule for when this
methodology is enough vs when manual mic verification is still required.

## The harness — `scripts/sweep_latency.py`

For each candidate `output_latency_ms`:

1. Patch `sd.InputStream` with `_FixtureInputStream` — a wall-clock-paced
   reader of `tests/fixtures/auto_sweep_input.wav` that mimics how a
   real mic delivers audio CONCURRENTLY with engine processing (not
   sequentially). The naive `time.sleep(frames/sr)` per `read()` adds
   inference time to every loop iteration and inflates engine cadence
   to `chunk_seconds + inference_ms`. The corrected version tracks
   wall-clock from `start()` and only sleeps as much as needed to keep
   `read()` returns aligned to a fixture cursor at sample rate.
2. Build an `EngineConfig` with the candidate `output_latency_ms` (and
   any other variable we're sweeping) and start the engine in a
   background thread.
3. Wait 4 s for engine warmup (cuDNN autotune + the 3 s lead-silence
   block in the fixture).
4. Reset the engine's cumulative stats counters (`xruns`,
   `queue_full_events`, `late_chunks`, `_recent_inference`,
   `chunks_processed`). This is required — the warmup window
   accumulates queue_full events when the playback backend has no
   downstream consumer to drain WoysSink.monitor.
5. Capture WoysSink.monitor for 60 s via `parec --device=WoysSink.monitor
   --rate=48000 --channels=2 --format=s16le --raw` → wrap as mono WAV.
6. Stop the engine, snapshot stats (xruns, queue_full, late_chunks,
   inference avg+p99+max, chunks_processed).
7. Run `woys-diag analyze <capture.wav> --no-spectrogram` and parse
   cuts/min from either the verdict line (`X events across Ds (Y/min)`)
   or the markdown report's `Events per minute` row.

Outputs:
- `docs/sweep_<timestamp>/capture_lat<N>_<backend>.wav` — raw capture per setting
- `docs/sweep_<timestamp>/capture_lat<N>_<backend>.md` — woys-diag report
- `docs/sweep_latency_<timestamp>.json` — full numeric results
- `docs/sweep_latency_<timestamp>.png` — latency-vs-cuts/min plot

## The fixture — `tests/fixtures/auto_sweep_input.wav`

60-second mono 48 kHz int16 PCM, deterministic seed (RNG=42). Composition
matches `woys-diag`'s 60 s protocol blocks exactly so the diag's per-block
statistics line up:

| Time   | Block          | Content                                                                |
|--------|----------------|------------------------------------------------------------------------|
| 0–3 s  | lead_silence   | exact zeros (stresses input gate / vocoder voicing-floor hallucination)|
| 3–13 s | vowel          | 200 Hz fundamental + 5 harmonics + 5 Hz vibrato (sustained voicing)    |
| 13–23 s| counting       | 0.30 s voiced bursts at varying f0 (110–280 Hz) with 0.20 s gaps       |
| 23–33 s| plosives       | 30 ms noise-burst + decay every 250 ms (boundary-phase stress)         |
| 33–43 s| fricative      | band-pass-filtered white noise, 4–8 kHz                                |
| 43–53 s| normal         | stochastic interleaved vowel + plosive + brief gap                     |
| 53–60 s| tail_silence   | exact zeros                                                            |

Built and regenerated by `scripts/gen_sweep_fixture.py`. Peak normalized
to −3 dBFS. Each block's RMS is reported on regen.

## What works

The harness reliably distinguishes:

- Engine timing failures (`late_chunks`, `chunks_processed`,
  inference avg/p99) from output-side failures (`xruns`,
  `queue_full_events`).
- Total wall-clock change as a function of (chunk_seconds,
  output_latency_ms) — the deterministic part.

The cadence fix (concurrent vs sequential mic delivery) was the
single biggest correctness bug; without it, every measurement at
small chunk_seconds looked catastrophic because the engine main loop
was running at `chunk_seconds + inference_ms` cadence and pacat / pw-cat
underran constantly. After the fix, enqueue intervals match
`chunk_seconds * 1000` to within ±5 %, matching real-mic behavior.

## What doesn't work — the synthetic-vs-real correlation gap

**`woys-diag analyze` cuts/min on synthetic-input captures over-counts
~5–7× compared to real-speech captures at the same `output_latency_ms`
setting.**

Measured during the rc2 sweep:

- Real-mic v0.6.10 at `output_latency_ms=300`: ~12 cuts/min (per
  `docs/13-detector-calibration.md`).
- Synthetic-fixture sweep at `output_latency_ms=300`: 51.7 cuts/min.

Across the whole sweep range (80–320 ms), synthetic cuts/min stayed
roughly flat at 78–105 events/min — the gap detector was firing on
RVC's natural output amplitude variations when fed synthetic
(out-of-distribution) input. The vocoder produces brief amplitude
dips inside its own output frames whenever its internal state
diverges from the speech-shaped manifold; those dips drop below the
detector's `SILENT_DBFS = -55 dBFS` threshold and get counted.

This is **not a buffer-underrun signal**. It's a model-output
artifact. The ratio is roughly constant across `output_latency_ms`,
so the harness can rank settings *relatively* but cannot directly
predict the absolute real-world cuts/min.

## The rule

> Use the harness to **rank settings** and to **rule out catastrophic
> regressions**. Use ONE real-mic test to **calibrate the absolute
> number** at the chosen setting.

Specifically, after a sweep:

1. Find the smallest `output_latency_ms` where the synthetic harness
   shows zero engine `queue_full_events` (the buffer-underrun signal
   that's NOT subject to the model artifact).
2. Add a safety margin equal to the engine's p99 inference jitter
   (p99 inference − chunk_seconds × 1000, bounded below at zero).
3. Get ONE real-mic woys-diag run at the chosen setting; compare its
   cuts/min against v0.6.10's 12/min ground truth.
4. If the real number is within tolerance (< 18/min was the rc2
   target), ship. If higher, bump output_latency_ms by 50 ms and
   re-test (the harness has already ruled out underruns at lower
   values, so a single targeted re-test is the right loop, not
   another sweep).

## v0.7.0-rc2 application

Applied the rule:

- Synthetic sweep showed cuts/min flat at 78–105 across 80–320 ms.
- Engine `queue_full_events` was zero everywhere (modulo the warmup
  artifact noted above) — the engine itself is not outpacing the
  writer at any of these settings; the variance is downstream.
- p99 inference at chunk=0.15 was 130–200 ms vs a 150 ms budget —
  occasional 50 ms overage. The output buffer must absorb at least
  this much. A 50 ms safety margin × 2 (rounded) = 100 ms.
- 80 ms (rc1) was insufficient and confirmed bad by user.
- 220 ms gives a clean 70 ms margin above the worst observed
  budget overage and lands cleanly in the flat region of the
  synthetic plot. Saves 80 ms vs v0.6.x's 300 ms default.

The rc2 ship gate is the single user mic test at 220 ms.

## Future use

When tuning any per-chunk timing parameter (chunk_seconds,
output_latency_ms, sola_*, prime_silence_seconds, an introduced
ring-buffer size), drive the sweep with this harness first. The
harness rules out structural failures in seconds; the user mic
verifies the perceptual ground truth in 30 s. Avoid the
seven-manual-tests pattern.

If a future change introduces a cadence-affecting knob (e.g. an
input-side prefetch), update `_FixtureInputStream` to model the new
behavior — wall-clock alignment is the load-bearing invariant, not
the specific sleep pattern.
